## 02 | 为什么会出现这个问题？

### 🎯 核心原因：**背景信息不对等**

每个行业都有自己的：
- **专业术语**（TAM/SAM、LTV/CAC、K8s、Service Mesh）
- **思维框架**（增长黑客、产品思维、技术架构）
- **关注重点**（转化率、用户体验、系统稳定性）
- **盲点**（不懂对方的约束条件）

### ❌ 传统解决方案

1. **请专家**：贵！（一小时几千元咨询费）
2. **学习新领域**：慢！（至少需要几个月）
3. **反复沟通**：累！（容易误解和冲突）

## 04 | 实战案例：3个典型场景

### 📋 案例1：技术架构决策

**场景**：创业公司CTO需要决定"用PostgreSQL还是MongoDB"

**传统方式**：
- Google搜索 → 一堆技术文章 → 看了半天还是不懂
- 问技术朋友 → 每个人答案不一样
- 拍脑袋决定 → 可能选错

**使用Cosplayer**：

```
用户："我们应该用PostgreSQL还是MongoDB？"

【角色切换：数据库架构师】

迁移评估框架：

1. 数据模型匹配度
   - PostgreSQL: 关系型、ACID、复杂查询
   - MongoDB: 文档型、灵活schema、嵌入式关联

2. 查询模式分析
   - 大量JOIN? → PostgreSQL更适合
   - 嵌套文档? → MongoDB更简单
   - 全文搜索? → 都需要额外工具

3. 迁移成本
   - 查询重写: [估算]
   - 数据迁移: [估算]
   - 风险: 数据一致性

【角色恢复：决策者】

建议：先分析查询模式，JOIN多留PostgreSQL，嵌套多考虑MongoDB
```

**效果**：5分钟得到专业分析，而不是几小时的盲目搜索。

```
   - Cookie同意管理
   - 隐私政策更新
   - 数据处理协议

第一步：做数据清单，第二步：咨询律师确认细节
```

**效果**：先有清晰框架，再针对性请律师（省时省钱）。

## 05 | 为什么不是所有问题都需要切换角色？

### ❌ 这些问题**不需要**切换

1. **简单编程问题**
   ```
   用户："Python怎么读JSON？"

   AI直接回答：
   import json
   with open('file.json', 'r') as f:
       data = json.load(f)
   ```
   **原因**：通用知识足够，不需要专家视角。

2. **概念性问题**
   ```
   用户："SQL和NoSQL有什么区别？"

   AI直接回答：
   SQL是关系型数据库，NoSQL是非关系型...
   ```
   **原因**：简单对比，不需要专业框架。

### ✅ 这些问题**需要**切换

1. **复杂决策**（技术架构、市场战略、投资决策）
2. **专业领域**（法律合规、财务分析、产品设计）
3. **冲突优先级**（速度vs质量、增长vs利润）
4. **你明确要求**特定视角（"从安全角度分析"）
